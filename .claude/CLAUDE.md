# Radiofy — Radio Playlist Sync

## Confidentiality

**IMPORTANT**: Do not claim AI authorship of any change. The repo is public and free to mention AI / LLM tools as *external tools the operator uses* (see `PROJECT_ARCHITECTURE.md → Manual Overrides → Authoring workflow`). The line to never cross is suggesting that the code itself was written by an AI assistant.

Forbidden — the pre-commit hook blocks these:

- Commit / PR / code text like `Generated by Claude`, `Written with Copilot`, `AI-generated`
- Co-author trailers crediting an AI tool (e.g. `Co-Authored-By: Claude <...>`)
- Emoji authorship markers (`🤖 Generated with ...`)

Allowed:

- Mentioning AI / LLMs as a tool an operator may use in a documented workflow
- Path / file references such as `.claude/CLAUDE.md`
- Discussion of "AI-assisted" features when describing future product capabilities

## Public Repository Hygiene

**IMPORTANT**: This repository is public. Everything committed — code, tickets in `docs/ticket/`, design docs, configuration templates, commit messages, PR text — is visible to anyone on the internet. Tickets live *inside* the repo and are pushed with the code; treat them with the same care as production code.

The following must **never** appear in any committed file:

- `SPOTIFY_CLIENT_SECRET`, refresh tokens, access tokens, `Bearer` strings (`.env` is gitignored; `.env.example` carries empty placeholders)
- Personal data: real names, emails, addresses, listening history
- Customer / employer names, internal URLs, internal product names
- Output of `bun run export-unmatched` (reveals listening patterns indirectly)
- Snapshots of `storage/db/*.db` or `storage/logs/*.log` (gitignored — never `-f` them in)

**OK to include** (these are identifiers, not secrets):
- Spotify playlist **names** in `config/stations.json` (the worker resolves names to IDs at runtime via the user's OAuth token — see architecture "Playlist discovery"). Forkers replace the names with their own and create matching playlists in their Spotify account.
- Spotify `client_id` (if ever inlined) — public per OAuth design; still kept in `.env` by convention
- Spotify track URIs in test fixtures or architecture examples — public catalog identifiers
- IDs from the public aggregator fixture (e.g. `utwor/86665`) — already public on the source site
- Synthetic example data clearly marked as such (use `<your-...>` placeholders for things meant to be replaced per-fork)

## Coding Standards

### Inline comments
**IMPORTANT**: Inline comments are NOT allowed and must be removed. Only structured doc comments (`/** ... */`) are permitted, and only where the *why* is non-obvious.

### Project rules
- One change = one ticket. No code without a ticket in `docs/ticket/backlog/`.
- Keep diffs minimal. No unrelated refactors, no opportunistic formatting changes, no comment churn on untouched code.
- **Deterministic logic first**: song matching, parsing, and normalization must be deterministic and testable. AI-based matching is not allowed as the primary path.
- **Cache aggressively**: Spotify search results are expensive and rate-limited — every track lookup must be cached in the database before hitting the network.
- **Crawling**: `fetch` + `cheerio` is the default. Playwright is a fallback, only when fetched HTML is incomplete or JS-rendered.
- **Strict TypeScript**: no `any` in production code; explicit return types on exported functions; no implicit `any` from `tsconfig`.
- **Boundaries first**: each package (`sources`, `normalizer`, `matcher`, `spotify`, `database`, `shared`) exports a narrow public surface. Internals stay internal.
- Business logic without tests is not done. Tests live next to their package.

## Code Quality Commands

### Code Style
Format and auto-fix a single file:
```bash
bunx biome check --write <file>
```

**IMPORTANT**: When multiple files changed, run the command per file (not multiple paths in one invocation).

### Lint / Types
```bash
bunx tsc --noEmit
```

### Tests
Two layers, separate directories:

- **Unit tests** live next to the package they cover: `packages/<pkg>/test/*.test.ts`. They never hit the network — Spotify and HTTP fixtures are stubbed.
- **Integration tests** live at the repo root in `tests/integration/*.test.ts`. They exercise the real Spotify API against a dedicated test playlist and are **env-gated**: skipped unless `RADIOFY_INTEGRATION=1` is set in the shell. CI never runs them.

All tests (unit + skip-gated integration):
```bash
bun test
```

Only unit tests:
```bash
bun run test:unit
```

Only integration tests (requires `RADIOFY_INTEGRATION=1` + a `.env` with valid Spotify credentials + an existing playlist named by `RADIOFY_INTEGRATION_PLAYLIST`):
```bash
RADIOFY_INTEGRATION=1 bun run test:integration
```

Single test file:
```bash
bun test packages/<pkg>/test/<name>.test.ts
```

### Quality Gate
After any change, run in this order:
1. `bunx biome check --write <file>` — fix style per file
2. `bunx tsc --noEmit` — type-check the whole project
3. `bun test` — run tests

**IMPORTANT**: Run the style fix on every changed file again right before committing. CI checks the whole project — including import order and formatting details that are easy to miss locally.

## Overview

Radiofy is a Bun + TypeScript worker that crawls Polish radio station playlist pages (ZET, ESKA, RMF MAXX, RMF FM), normalizes the extracted songs, deduplicates them across the week, matches them against Spotify's catalogue and replaces the contents of one persistent Spotify playlist per station.

Stack: TypeScript / Bun runtime (no application framework)

Source of truth for architecture: `docs/architecture/PROJECT_ARCHITECTURE.md`

## Workflow

### Ground rules
1. Read this `CLAUDE.md` before starting any task.
2. **Always work from a backlog ticket. No changes without a ticket — including bug-fixes, follow-ups, docs, and one-line changes.** If the change does not yet have a ticket, write one first (`.claude/templates/ticket-template.md`) and put it under `docs/ticket/backlog/in-progress/`.
3. Only modify files that belong to the active ticket. No unrelated changes.
4. Keep diffs minimal. No refactoring, formatting, or comments on code not part of the ticket.

### Implementer fast path
For small fixes the implementer may write, implement, and test the ticket in one pass, then move it **directly to `review/`** (skipping the manual `/ticket-start` formality). The reviewer is still the gate to `done/`. The order is:

1. Write the ticket file in `docs/ticket/backlog/in-progress/<id>_<slug>.md` (use the template).
2. Branch (`rdfy-<id>-<slug>`), implement, write tests, run the Quality Gate.
3. `git mv` the ticket from `in-progress/` to `review/`.
4. Commit the work and the ticket move together, push, open the PR.
5. Dispatch the reviewer agent. If approved, the reviewer moves it to `done/` and writes the result document; the implementer commits that and merges.

The slow path (`/ticket-start` → human pause → resume) still applies when the scope is unclear or the change is high-risk and benefits from an explicit alignment step before code is written.

### Ticket flow
Tickets live in `docs/ticket/backlog/<status>/`, filename `<ticket-id>_<slug>.md`.
Template: `.claude/templates/ticket-template.md`
Sample data and fixtures for tests: `docs/ticket/testdata/`.

```
todo → in-progress → review → done
```

1. **todo**: Ticket exists, not yet started.
2. **in-progress**: Implementer pulls the ticket from `todo/` and moves it to `in-progress/`. Work begins.
3. **review**: Implementer moves the finished ticket to `review/`. Reviewer checks it.
4. **done**: Reviewer moves the verified ticket to `done/`.

The ticket file is *moved*, not copied. Only one ticket in `in-progress` per owner at a time.

Folder transitions are automated by the slash commands `/ticket-start`, `/ticket-review`, and `/ticket-done`.

### Agents

- **reviewer**: Checks implementations against ticket criteria and security
